## What v0.3.0 Delivered

### §1 Core-Goal Invariant (spec §14)

Every `BasicAbility` profile now requires a top-level `core_goal` block
with three fields:

* `statement`: prose declaration of the user-observable function.
* `test_pack_path`: pointer to a YAML pack of ≥ 3 Pact-style cases.
* `minimum_pass_rate`: spec-locked at `1.0` (not configurable).

The runtime computes `C0_core_goal_pass_rate = cases_passed / cases_total`
every round; the **§14.3.1 universal MUST**: `C0 = 1.0` regardless of
round_kind. The **§14.3.2 no-regression rule**: `C0(round_n) ≥
C0(round_n-1)` — any C0 < prior_round = ROUND FAILURE regardless of
other axes; rollback-only response per §14.4. C0 is a **top-level
invariant** (NOT an R6 D8 sub-metric) — this preserves the frozen
7×37 R6 count exactly.

Si-Chip's own `core_goal_test_pack.yaml` (Round 14 first authoring, 5
cases, pack_version 1.0.0) covers the four new v0.3.0 generic tools
(`eval_skill --help` + `round_kind validate {4 kinds}` +
`spec_validator --json` dual-spec + `count_tokens` packaging gate) and
the §8.2 6-evidence-file completeness check.

### §2 round_kind Enum (spec §15)

Every `next_action_plan.yaml` and `iteration_delta_report.yaml` now
declares `round_kind ∈ {code_change | measurement_only | ship_prep |
maintenance}` with a per-kind `iteration_delta` clause:

| `round_kind` | iteration_delta clause | Allows source change? |
|--------------|------------------------|-----------------------|
| `code_change` | strict (≥ 1 axis at gate bucket) | YES |
| `measurement_only` | monotonicity_only (no axis required to improve, none may regress) | NO |
| `ship_prep` | WAIVED (release packaging only; no metric movement required) | YES (version bumps + tarball) |
| `maintenance` | WAIVED (housekeeping; no metric movement required) | YES (no functional change) |

Universal C0 = 1.0 + monotonicity holds across ALL 4 kinds (§15.3).
The `consecutive-rounds` promotion rule (§15.4) advances regardless
of round_kind, provided per-kind eligibility (a) C0 = 1.0; (b) C0 ≥
prior C0; (c) no v1_baseline regression — Round 14 + Round 15 both
satisfy this.

This **resolves the v0.2.0 measurement-fill pain**: in v0.2.0 cycle
Rounds 4-12 used "measurement-fill" flavours masquerading as
`code_change` rounds (per v0.2.0 ship_report §"What Round 13
Delivered" §3); v0.3.0 separates the two via §15.1 enum + §15.2 clause.

## Why v2_tightened Is Deferred to v0.4.0

`spec §4.2 promotion rule` requires CONSECUTIVE TWO ROUNDS clearing
every v2_tightened hard threshold. Round 14 + Round 15 v2 trace:

| Threshold | Round 14 | Round 15 |
|-----------|---------:|---------:|
| `pass_rate ≥ 0.82` | 0.85 PASS | 0.85 PASS (replay) |
| **`pass_k ≥ 0.55`** | **0.5478 FAIL** | **0.5478 FAIL** (carry-forward; structural) |
| `trigger_F1 ≥ 0.85` | 0.8934 PASS | 0.8934 PASS (replay) |
| `near_miss_FP_rate ≤ 0.10` | 0.05 PASS | 0.05 PASS |
| `metadata_tokens ≤ 100` | 97 PASS | 97 PASS |
| `per_invocation_footprint ≤ 7000` | 3710 PASS | 3710 PASS |
| `wall_clock_p95 ≤ 30 s` | 1.4693 PASS | 1.4693 PASS |
| `routing_latency_p95 ≤ 1200 ms` | 1100 PASS | 1100 PASS |
| `routing_token_overhead ≤ 0.12` | 0.0233 PASS | 0.0233 PASS |
| **`iteration_delta any axis ≥ +0.10`** | +0.05 FAIL (v1 bucket only) | 0.0 RELAXED (per §15.2 `monotonicity_only` clause for `measurement_only`) |

`consecutive_v2_passes = 0`. The blocker is the SAME structural one as
v0.2.0: the deterministic SHA-256 simulator emits `pass_k_4 =
pass_rate^4` PROXY (a lower bound). Real k=4 sampling against an actual
LLM typically produces real `pass_k_4 > mean^k` because variance pushes
some runs above the mean. This is independent of which features land
in v0.3.0 — adding eval cases cannot lift the proxy because the
formula averages-in their proxy values.

The genuine unblock is `evals/si-chip/runners/real_llm_runner.py`
which does k=4 independent sampling per prompt against composer_2 /
sonnet / opus. The v0.2.0 ship report titled this "Roadmap to v0.3.0"
because v0.2.0 ship-time L0 directive was `PATH=REVERT-ONLY` (sandbox
lacks LLM API auth + outbound https). v0.3.0 was scoped instead to
the smaller-footprint, sandbox-executable items: core_goal invariant,
round_kind enum, and the multi-ability layout. Therefore the same
section is renamed **"Roadmap to v0.4.0"** for this ship.
