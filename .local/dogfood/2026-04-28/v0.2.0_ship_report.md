## Path A (recommended) — Real-LLM runner upgrade

The `T2_pass_k` blocker is a structural artifact of the deterministic
SHA-256 simulator. The simulator emits `pass_k_4 = pass_rate^4` as a
documented PROXY (a lower bound — the real expected k=4 sample is
typically higher than `mean^k` because variance pushes some runs
above the mean).

Round 13 (real-LLM runner) + Round 14 (v2 verification) would:

1. Replace `evals/si-chip/runners/with_ability_runner.py` per-prompt
   simulator with real LLM invocations (e.g. composer_2 + sonnet
   thinking depths) and produce a real `pass_k_4` by sampling k=4
   independent runs per prompt.
2. Naturally close G1 partial-proxy + L5/L6/L7 + R1/R2 + the §6.4
   trigger 2/5/6 catalog gaps (scenario catalog + wrapper-stability
   log + invocation log can be wired alongside).
3. Verify two consecutive v2_tightened passes via the §4.2 promotion
   rule.

This is the v0.3.x roadmap item already deferred in
`round_11/next_action_plan.yaml` — Round 12 BRINGS IT FORWARD as the
canonical path A.
